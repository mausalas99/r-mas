# R+ 3.0.0 — Modos Sala / Interconsulta, Estado Actual, Listado de Problemas y DOB

**Fecha:** 2026-05-07
**Versión objetivo:** R+ 3.0.0 (mayor — cambia el modelo del paciente)
**Status:** Diseño aprobado, listo para plan de implementación

## Resumen

R+ 3.0.0 introduce el concepto de **modo de trabajo** del usuario: el expediente cambia su forma según seas médico de Sala (atención hospitalaria continua, todos en Medicina Interna) o de Interconsulta (atención puntual a pacientes de otros servicios).

- **Sala** oculta Nota de Evolución e Indicaciones, expone un botón **Estado Actual** (snapshot rápido del paciente) y una pestaña nueva **Listado de Problemas** (con generador `.docx`).
- **Interconsulta** mantiene la app como hoy.

Adicionalmente, el modelo de paciente añade **Fecha de Nacimiento** como campo requerido para nuevos pacientes; la edad pasa a calcularse desde DOB.

El modo se elige una sola vez en Mi Perfil → Aplicación. **Default = Sala** (caso de uso primario actual). Usuarios existentes ven un toast informativo en su primer arranque de 3.0.

## Motivación

Hoy R+ asume un único flujo: cada paciente puede tener nota, indicaciones, tendencias y cultivos. En la práctica clínica de Sala (Medicina Interna), la nota completa y la hoja de indicaciones son menos relevantes que un *snapshot* rápido del estado del paciente y una lista mantenida de problemas activos/inactivos. El modelo único obliga a navegar tabs que no se usan y a no tener herramientas que sí necesitan.

La separación por modo permite especializar la UI y los entregables sin duplicar código pesado, y permite extender a futuros modos (UCI, Urgencias) si surge la necesidad.

## Decisiones de diseño

### Decisión: implementación del modo con `if/else` simple

Evaluadas tres opciones (capabilities table, if/else simple, componentes separados). El usuario eligió `if/else` simple — `settings.appMode === 'sala'` checked en cada call site. Trade-off aceptado: más call sites con la condición, pero cero abstracción extra. Si en el futuro se agrega un tercer modo, se refactoriza entonces.

Implementación: helper `isModeSala()` (lee setting) usado en cada lugar relevante para mantener consistencia y permitir un cambio futuro sin tocar todos los sitios. Los call sites se ven como:

```js
if (isModeSala()) {
  // ocultar Nota
}
```

### Decisión: Estado Actual reusa modal SOAP existente, oculta `S — Subjetivo`

En vez de construir un modal nuevo, el modal `#soap-modal-backdrop` se reusa con renderizado condicional:

- Header cambia: "Plantilla de Evolución" → "Estado Actual" cuando se invoca desde el botón de Sala.
- Sección `S — Subjetivo` oculta con `display:none` (no removida del DOM, para preservar binding y permitir reactivar fácilmente).
- Botones del footer cambian: "Insertar en evolución" → "Guardar y copiar"; se añade botón secundario "Solo copiar (sin guardar)".

`buildSOAPText()` se ajusta para omitir la parte `S` cuando la sección está oculta. La salida del modal en Sala persiste en `notes[id].estadoActual` y `notes[id].estadoActualHora`, y se copia al portapapeles.

### Decisión: pestaña Listado de Problemas paralela a Indicaciones

El Listado se modela como una colección paralela a `indicaciones`, con el mismo patrón de auto-save por `oninput` y filas dinámicas (similar a "Otros" de indicaciones). Razón: reusar patrones probados (drag-and-drop, persistence, render) en vez de inventar una arquitectura nueva. Sin límite numérico (el template original muestra 1, 2, 3 numerados pero el python script clona filas).

### Decisión: DOB como fuente de verdad, edad como caché derivado

El paciente almacena `fechaNacimiento` (ISO `YYYY-MM-DD`) y `edad` (string display). En `commitPatient`, `edad` se setea desde `calculateAge(dob).display`. Razón: minimizar el cambio en el resto del código (filtros, tendencias, render) que ya leen `patient.edad`. Cada vez que se actualiza el paciente, se recalcula edad. Pacientes legacy sin DOB conservan su `edad` string original.

### Decisión: default = Sala

Para el caso primario actual (médico de Medicina Interna) la default es Sala. Usuarios existentes que abren 3.0 por primera vez son flipeados a Sala con un toast informativo (`"R+ 3.0 — Sala activado por defecto. Cambia el modo en Mi Perfil → Aplicación"`). Trade-off: friction visual leve la primera vez, pero alineado con el caso primario. Reversible en 1 clic.

## Modelo de datos

### Settings (`localStorage 'rpc-settings'`)

```js
{
  // ... campos existentes ...
  appMode: 'sala' | 'interconsulta',     // nuevo, default 'sala'
  defaultServicio: string,                // nuevo, default '' hasta tutorial
  _v3MigrationDone: true                  // flag interno de migración
}
```

### Patient

```js
{
  id, nombre, registro,
  fechaNacimiento: 'YYYY-MM-DD' | null,   // nuevo
  edad: string,                            // existente; ahora derivado de DOB para nuevos
  sexo, area, servicio, cuarto, cama,
  fromLab, archived, pinned,
  // ... otros campos existentes ...
}
```

### Notes (existente, dos campos nuevos)

```js
notes[patientId] = {
  // ... campos existentes (interrogatorio, evolucion, etc.) ...
  estadoActual: string | null,             // nuevo: snapshot SOAP-sin-S
  estadoActualHora: 'DD/MM/YYYY HH:MM' | null  // nuevo
}
```

### Listado de Problemas (nueva colección)

```js
listadoProblemas[patientId] = {
  fecha: 'DD/MM/YYYY',
  hora: 'HH:MM',
  activos:   [{ id, fecha: 'DD/MM/YYYY', descripcion: string }, ...],
  inactivos: [{ id, fecha: 'DD/MM/YYYY', descripcion: string }, ...]
}
```

`id` por problema = timestamp+random (mismo patrón que pacientes y "Otros" de indicaciones), para soportar drag-and-drop y borrado posicional.

### Persistencia

`storage.js` añade:

- `getListadoProblemas() / saveListadoProblemas()`
- Inclusión de `listadoProblemas` en `saveAll(...)`
- Inclusión de `listadoProblemas` en exportación / importación de respaldo JSON completo.

### Helper de edad

```js

  // o null si dob inválido (futuro, >120 años, no parseable)
}

function formatDobForDocs(dob) {
  // 'YYYY-MM-DD' → 'DD/MM/YYYY' para insertar en plantillas .docx
}
```

**Timezone:** ambos helpers parsean `dob` como fecha civil (no UTC instante). El cálculo usa `Date.UTC(year, month-1, day)` y `Date.UTC` del `asOf` truncado a día, para que un usuario en zona horaria distinta no vea fluctuar la edad por una hora. Tests cubren cambios de día y bordes de año/mes.

Cobertura: tests unitarios `age-calc.test.mjs`.

### Helper de modo

```js
function isModeSala() {
  return (settings.appMode || 'sala') === 'sala';
}

function getDefaultServicio() {
  return (settings.defaultServicio || '').trim();
}
```

Tests: `mode-features.test.mjs` con casos para missing setting, valor inesperado, etc.

### Modal de Nuevo Paciente

Cambios al modal `#modal` (en `public/index.html`):

- Campo nuevo **Fecha de nacimiento** = `<input type="date">`. Visible en variantes manual y from-lab.
- Campo **Edad** pasa a read-only display (`<span>` o `<input readonly>`) — actualizado por listener `oninput` del DOB con `calculateAge(...).display`.
- Validaciones (en `savePatient`):
  - `dob` requerido (toast `"Ingresa la fecha de nacimiento"` y focus al campo).
  - `dob` no puede ser futuro.
  - `dob` no puede ser >120 años atrás.
- Pre-llenado de **Servicio** = `getDefaultServicio()` cuando el modo es Sala.
- Variante "desde lab": muestra hint `"📋 Lab dice: 58 años"` como referencia visual; el guardado sigue exigiendo DOB.

Patient legacy (sin DOB) editado: el campo aparece vacío con badge ámbar `"Faltante"`. Save NO obliga llenar DOB para no romper datos viejos. Documentos generados (nota, listado) usan fallback al string `edad` cuando DOB falta.

### Tabs condicionales del expediente

`renderInnerTabs()` (existente o nuevo helper) lee `isModeSala()` y aplica `display:none` a las tabs ocultas. Lista por modo:

| Tab DOM ID | Interconsulta | Sala |
| --- | --- | --- |
| `itab-notas` | visible | hidden |
| `itab-indica` | visible | hidden |
| `itab-tend` | visible | visible |
| `itab-cult` | visible | visible |
| `itab-listado` (**nuevo**) | hidden | visible |

Default tab al abrir paciente:
- Interconsulta: `notas` (igual que hoy).
- Sala: `tend` (Tendencias).

Si el modo cambia con paciente activo en una tab que el nuevo modo oculta → redirigir a default del nuevo modo.

### Botón "Estado Actual" (Sala)

Botón superior en el header del expediente, al lado de Salida Rápida:

```html
<button id="btn-estado-actual" onclick="openEstadoActualModal()">
  📋 Estado Actual
  <span class="badge-hora" id="estado-actual-badge"></span>
</button>
```

`openEstadoActualModal()` reusa `openSOAPModalDirect()` con flag `mode='estado'`:

- Aplica `body.dataset.estadoActualMode = 'true'` para que el CSS oculte la sección `S` y el botón footer cambie de label.
- Si `notes[activeId].estadoActual` ya existe, el botón muestra badge con la hora (`estadoActualHora`).
- Hover/click cuando hay snapshot previo: tooltip con preview + opción "Ver" / "Capturar nuevo".

### Modal SOAP — modo Estado Actual

CSS condicional:

```css
body[data-estado-actual-mode="true"] #soap-modal-backdrop .soap-section.s-subjetivo {
  display: none;
}
body[data-estado-actual-mode="true"] #soap-modal-backdrop .soap-modal-footer .btn-insertar {
  display: none;
}
body[data-estado-actual-mode="true"] #soap-modal-backdrop .soap-modal-footer .btn-guardar-copiar {
  display: inline-flex;
}
body[data-estado-actual-mode="true"] #soap-modal-backdrop .soap-modal-footer .btn-solo-copiar {
  display: inline-flex;
}
```

Botones nuevos en footer del modal (siempre presentes, visibilidad por CSS):

- **Insertar en evolución** (Interconsulta) — comportamiento actual.
- **Guardar y copiar** (Sala) — `notes[activeId].estadoActual = buildSOAPText()`, `estadoActualHora = now()`, `navigator.clipboard.writeText(...)`, `saveState()`, toast.
- **Solo copiar (sin guardar)** (Sala) — `navigator.clipboard.writeText(buildSOAPText())`, toast, no persiste y **no sobreescribe** un Estado Actual previo.

`buildSOAPText()` se modifica para omitir la parte `S` cuando el modal corre en `data-estado-actual-mode`. El texto resultante en Sala arranca directamente con `N: ...` (no `S:` al inicio). Los separadores `||` entre secciones se mantienen idénticos al output actual; solo cambia el conjunto de secciones incluidas.

**Persistencia y cambios de modo:** `notes[id].estadoActual` y `estadoActualHora` se conservan al cambiar a Interconsulta y al regresar a Sala. En Interconsulta no hay UI que los lea, pero los datos no se borran. Si el médico borra el texto desde una sesión Sala, se pone a `null`.

### Generador `.docx` de Listado de Problemas

Archivos nuevos:

- `template_listado.docx` — copia del archivo `LISTADO DE PROBLEMAS LIMPIO (1).docx` movido a la raíz del repo.
- `generate_listado.py` — paralelo a `generate_note.py` y `generate_indicaciones.py`.

`generate_listado.py` (stdlib only, mismo patrón):

1. Lee JSON de stdin:
   ```json
   {
     "patient": { "nombre", "registro", "edad", "fechaNacimiento", "sexo", "area", "servicio", "cuarto", "cama" },
     "listado": { "fecha", "hora", "activos": [...], "inactivos": [...] },
     "doctor": { "doctorName", "grado" }
   }
   ```
2. Abre `template_listado.docx` como zip.
3. Reemplaza placeholders del header (las **dos** apariciones, igual que nota):
   - `NOMBRE: ` ← `nombre`
   - `REGISTRO: ` ← `registro`
   - `EDAD: ` ← `edad`
   - `FECHA DE NACIMIENTO: ` ← `formatDobForDocs(fechaNacimiento)` (o vacío si null)
   - `SEXO: ` ← `sexo`
   - `MEDICINA INTERNA` ← `servicio` (literal en el template; se reemplaza por el servicio real del paciente)
   - `CUARTO: ` y `CAMA: ` ← respectivos.
4. Tabla de problemas: detecta el `<w:tr>` placeholder con "1" en columna numérica y lo clona N veces. Cada fila clonada recibe número (`1, 2, 3, ...`), fecha, descripción del activo. Misma operación con la columna inactivos.
5. Firma: si `doctor.doctorName` no vacío, reemplaza la firma hardcoded `"R1 GC/MI JORGE ADRIÁN MONTEMAYOR CANTÚ"` por el nombre del usuario (formato `<grado> <nombre>` si grado existe).
6. Escribe el `.docx` resultante a stdout.

Endpoint nuevo en `server.js`:

```js
app.post('/generate-listado', (req, res) => {
  // mismo patrón que /generate-note y /generate-indicaciones
  // spawn python con generate_listado.py, pipe stdin, buffer stdout, set headers
});
```

Cliente: nueva función `generateListado(patientId)` en `app.js` que arma el JSON desde `patients[id]`, `listadoProblemas[id]` y `settings`, llama al endpoint, recibe blob, dispara descarga (o muestra modal de guardar) — mismo patrón que `generateNoteDocx`.

### Salida rápida según modo

Función existente `quickExportPatient(patientId)` se ramifica:

- Si `isModeSala()` → exporta listado de problemas si hay datos. Si no hay datos, toast `"Captura al menos un problema antes de exportar"` y aborta.
- Si Interconsulta → exporta nota (igual que hoy).

### Migración de localStorage

Función `migrateToV3()` corre en `storage.js` durante la inicialización:

```js
function migrateToV3() {
  const settings = getSettings();
  if (!settings._v3MigrationDone) {
    if (settings.appMode == null) settings.appMode = 'sala';
    if (settings.defaultServicio == null) settings.defaultServicio = '';
    settings._v3MigrationDone = true;
    saveSettings(settings);
    return true;
  }
  return false;
}
```

Si la migración corre, en `app.js` después de inicializar el state se dispara un toast informativo (única vez, controlado por el flag).

### Edge cases

1. **Cambiar modo con paciente activo en tab que el nuevo modo oculta:** detector en el handler del cambio de modo redirige a default del modo nuevo (Sala→tend, Interconsulta→notas).
2. **Generar listado con cero problemas:** botón generar valida `activos.length + inactivos.length > 0` antes de invocar; si no, toast bloqueante.
3. **Estado Actual sin captura previa:** modal abre vacío, comportamiento normal del modal.
4. **Lab con paciente nuevo:** modal de creación exige DOB aún en flujo from-lab; muestra hint con la edad parseada del lab como referencia.
5. **Borrar paciente:** `removePatient` también borra `listadoProblemas[id]` y, en `notes[id]`, los campos `estadoActual*` se borran al borrar el `notes[id]`.
6. **Exportar/Importar respaldo:** `exportBackup` incluye `listadoProblemas`, `settings.appMode`, `settings.defaultServicio`. `importBackup` los restaura. Test de round-trip.

## Versionado y compatibilidad

- **Versión:** 3.0.0 (mayor — DOB requerido en pacientes nuevos rompe la forma del modelo del paciente; modo cambia el conjunto de tabs visible).
- **min-version.json:** sin cambios; no se fuerza actualización.
- **Migración:** automática en primer arranque; idempotente; un solo flag.
- **Pacientes legacy:** intactos. DOB opcional para edición; documentos generados usan fallback a `edad` string si DOB falta.
- **Backups JSON anteriores a 3.0:** importables; los pacientes importados sin DOB quedan como legacy (DOB null, edad preservada).
