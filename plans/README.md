## Findings considered and rejected

So nobody re-audits these:

- `execSync` in `portInUseProcessHint` (`server.js:277`): only ever called with the constant `PORT`; the `ps -p ${pid}` arg comes from `lsof` output. Pattern-level nit, not a vulnerability.
- `/beacon` exposing `shiftPinActive` (`auth-router.js:35-43`): discovery UX — clients need to know whether to prompt for a shift PIN. Boolean only, no roster data. By design.
- Shift-PIN multi-use (no burn-on-exchange): by design — it's a monthly team credential many residents exchange.
- Rate limiting on `/auth/shift-pin` GET/regenerate: already behind bearer auth; the global limiter suffices.
- Dual sync+async write in `persistAlignedTeamCodeHash` (`host-store.js:213-217`): deliberate, commented mitigation against a prior store instance's queued snapshot clobbering the new hash.
- "Races" in `purgePatientFromAllRoomBundles` / `ensureLoadedSync`: synchronous single-threaded code with no `await` inside the critical sections — no interleave point exists.
- `labs.js` "has no tests": wrong — `procesarLabs` has golden characterization tests (`labs-procesar-characterization.test.mjs`) and ~20 `labs-*.test.mjs` suites cover the domain. Tier-2 legacy, decompose-when-touched per the ratchet.
- Electron 41 → 42 migration: no EOL/security pressure at audit time; monitor.
- `npm audit` findings: devDependency-only (tar chain via @electron/rebuild, minimatch via eslint-plugin-sonarjs); `npm audit --omit=dev` is clean. Known/monitor-only per the phase-5 hardening notes. Small optional cleanup: bump eslint-plugin-sonarjs to v4 when convenient.
- LAN plaintext HTTP, CSP `unsafe-inline`: documented risk acceptances (`docs/core/15-security.md`); TLS/RBAC are LATER-horizon items in the vision doc.
- `transport.mjs` globalThis deps dual-source: LOW-confidence, depends on bundle chunk layout; plan 008 pins its current behavior with tests instead of changing it.
- Snapshot-test quality, pre-commit hooks, architecture-doc diagram nit: real but below the leverage bar this cycle.
